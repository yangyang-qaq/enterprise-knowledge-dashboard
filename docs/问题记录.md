## 问题 1：Node.js 版本不兼容

**日期**：2026-07-20

**现象**：`npm install` 报错
```
npm error EBADENGINE
npm error Not compatible with your version of node/npm: open-webui@0.10.2
npm error Required: {"node":">=18.13.0 <=22.x.x"}
npm error Actual:   {"node":"v24.15.0"}
```

**原因**：本地 Node.js v24 超过了项目要求的 v22 上限，npm 新版默认强制检查 engine 版本。

**解决方案**：
```bash
npm install --engine-strict=false
```
v24 API 向下兼容，实际运行无任何问题。

**教训**：大型开源项目 package.json 的 engine 限制不一定及时更新，遇到类似问题先尝试 `--engine-strict=false`。

## 问题 2：WEBUI_SECRET_KEY 未设置导致启动失败

**日期**：2026-07-20

**现象**：直接运行 uvicorn 启动后端报错：
```
WEBUI_SECRET_KEY is not set. It is a hard requirement when authentication is enabled.
```

**原因**：项目在认证模式下要求必须设置 `WEBUI_SECRET_KEY` 环境变量。Docker 部署时自动生成，手动启动需要自己设置。

**解决方案**：
```bash
WEBUI_SECRET_KEY="$(python -c 'import secrets; print(secrets.token_hex(32))')" \
python -m uvicorn open_webui.main:app --host 127.0.0.1 --port 8080
```

**教训**：读开源项目代码时要关注环境变量依赖，`.env.example` 里不一定写了所有必需的变量。

## 问题 3：HuggingFace 模型下载失败（国内网络）

**日期**：2026-07-20

**现象**：服务启动后卡在 `Fetching 30 files`，最终报错：
```
Cannot determine model snapshot path: An error happened while trying to 
locate the file on the Hub and we cannot find the requested files in the 
local cache.
```

**原因**：huggingface.co 在国内被墙，默认的 sentence-transformers 模型（all-MiniLM-L6-v2）无法下载，导致应用启动阻塞在 lifespan 事件中，整个服务不可用。

**解决方案**：设置 HuggingFace 镜像：
```bash
HF_ENDPOINT="https://hf-mirror.com" python -m uvicorn ...
```

**教训**：在国内做 AI 开发，HuggingFace、GitHub 等资源都要考虑网络问题，提前准备镜像方案。

## 问题 4：代码修改后前端不生效

**日期**：2026-07-20

**现象**：修改 Svelte 组件后刷新浏览器看不到变化。

**原因**：FastAPI 以静态文件方式 serve `build/` 目录的前端产物。修改源码后必须重新 `npm run build` 才会更新 `build/` 目录。

**解决方案**：
```bash
npm run build        # 重新构建前端
# 然后重启后端
```

**教训**：SvelteKit adapter-static 模式下是纯静态导出，没有 HMR。开发时可以用 `npm run dev` 开启 Vite 开发服务器获得热更新，但需要额外配置 CORS 代理。

## 问题 5：Python 模块找不到 `ModuleNotFoundError: No module named 'open_webui'`

**日期**：2026-07-20

**现象**：在项目根目录执行 `uvicorn open_webui.main:app` 报模块找不到。

**原因**：`open_webui` 包在 `backend/` 目录下，不在 Python 默认的搜索路径中。

**解决方案**：必须从 `backend/` 目录启动：
```bash
cd backend
python -m uvicorn open_webui.main:app --host 127.0.0.1 --port 8080
```

**教训**：Python monorepo 项目的包路径依赖工作目录，不要随便换目录启动。

## 问题 6：端口被占用 `[winerror 10048]`

**日期**：2026-07-20

**现象**：重启服务时报错：
```
error while attempting to bind on address ('127.0.0.1', 8080): 
[winerror 10048] 通常每个套接字地址只允许使用一次
```

**原因**：上一次的后台进程没有完全退出，端口 8080 仍被占用。

**解决方案**：
```bash
# 查找占用端口的进程
netstat -ano | grep ":8080" | grep LISTENING
# 杀掉进程
taskkill //PID <进程ID> //F
```

**教训**：后台启动 uvicorn 后关终端不会自动杀进程，需要手动清理。写个启动脚本自动检测端口会更好。

## 问题 7：Tab 导航不跨页面显示

**日期**：2026-07-20

**现象**：Tab 导航放在 `[id]/+page.svelte` 里，切到 `/chunks` 页面后 Tab 消失了。

**原因**：SvelteKit 的 `+page.svelte` 只作用于当前路由。子页面 `/chunks` 有自己独立的 `+page.svelte`，不会继承父页面的 UI。

**解决方案**：改用 `+layout.svelte`，通过 `<slot />` 让 Tab 导航在所有子页面共享：
```
[id]/
  +layout.svelte     ← Tab 导航 + <slot />
  +page.svelte       ← Files 视图
  chunks/+page.svelte ← Chunks 视图
```

**教训**：SvelteKit 的 layout 机制是共享 UI 的正确方式。page 只渲染自身内容，跨页面共享的 UI（导航栏、侧边栏）必须放在 layout 里。

## 问题 8：`add_file_to_knowledge_by_id` 中进度钩子没触发

**日期**：2026-07-20

**现象**：通过 Web UI 上传文件到知识库后，Processing 页面始终显示 "No processing tasks yet"。

**排查过程**：
1. 检查前端 API 调用 → 正常，`/progress` 返回 `[]`
2. 检查数据库 → `knowledge_processing_task` 表为空
3. 检查 `add_file_to_knowledge_by_id` 端点 → 确认有 `_update_processing_progress` 调用
4. 抓前端上传时的网络请求 → 发现上传走的是 `/api/v1/files/` 端点，不是 `/api/v1/knowledge/{id}/file/add`

**原因**：Web UI 上传文件时，流程是：
```
POST /api/v1/files/ (带 metadata.knowledge_id)
  → process_uploaded_file() 
    → 内部直接调用 process_file() + Knowledges.add_file_to_knowledge_by_id()
    → 绕过了 routers/knowledge.py 中的进度钩子
```
我的进度追踪代码只加在了 `routers/knowledge.py` 的 `add_file_to_knowledge_by_id` 函数里，但 Web UI 上传流程根本不走这个端点，而是走 `routers/files.py` 的内部逻辑。

**解决方案**：在 `routers/files.py` 的 `process_uploaded_file()` 中也加入同样的进度追踪代码。

**教训**：在已有项目上做增量开发时，**必须追踪完整的数据流**，不能假设用户只通过某个 API 端点操作。同一个功能可能有多个入口（API 调用、Web UI 上传、批量导入），每个入口都要覆盖到。

## 问题 9：`merge_docs_to_target_size` 导入路径错误

**日期**：2026-07-20

**现象**：调用 `/chunks/preview` API 返回 500：
```
cannot import name 'merge_docs_to_target_size' from 'open_webui.retrieval.utils'
```

**原因**：`merge_docs_to_target_size` 函数定义在 `routers/retrieval.py` 中，但我在 `chunks/preview` 端点里写成了 `from open_webui.retrieval.utils import merge_docs_to_target_size`。

**解决方案**：改为正确路径：
```python
from open_webui.routers.retrieval import merge_docs_to_target_size
```

**教训**：不要在没验证的情况下假设函数的 import 路径。用 `grep` 确认函数实际定义位置后再写 import。

## 问题 10：SQLite WAL 模式下的数据恢复

**日期**：2026-07-20

**现象**：直接 DELETE 数据库中的用户数据后，想恢复时发现没有 `.bak` 备份文件。

**原因**：SQLite WAL（Write-Ahead Log）模式下，数据修改可能还在 WAL 文件中未完全合并到主 DB 文件。执行 `PRAGMA wal_checkpoint(TRUNCATE)` 可以强制合并。

**教训**：操作数据库前先备份，尤其是生产环境。SQLite 虽然有 WAL 保护，但不能替代备份。

## 问题 11：`query_collection` 参数名和返回值格式不对

**日期**：2026-07-20

**现象**：调用 evaluate/query API 时报错：
```
query_collection() got an unexpected keyword argument 'query'
```

**原因**：`query_collection()` 的实际签名是 `(request, collection_names, queries, embedding_function, k)`，其中 `queries` 是 list[str] 且必须传入 `embedding_function`。而我写成了 `query=form_data.query, user=user`。

修复参数后仍返回空结果，进一步发现返回值格式也不同：返回的是 dict 格式 `{'documents': [[]], 'distances': [[]], 'metadatas': [[]]}`，没有 `ids` 字段。

**解决方案**：
```python
# 正确调用
result = await query_collection(
    request=request,
    collection_names=[id],
    queries=[form_data.query],       # list, not str
    embedding_function=request.app.state.EMBEDDING_FUNCTION,  # required
    k=k,
)
# 正确解析
docs = result.get('documents', [[]])[0]
dists = result.get('distances', [[]])[0]
```

**教训**：调用开源项目内部函数时，不要凭经验猜参数和返回值。先用 grep 找到函数定义，确认签名和返回格式。

**日期**：2026-07-20

**原因**：curl 在 Windows Git Bash 下发送包含中文的 JSON 时，编码处理不稳定。

## 问题 13：`restored_chunks` 变量未定义导致 `NameError`

**日期**：2026-07-20

**现象**：rollback API 返回 500，响应体为空。Python 独立脚本测试正常。

**原因**：简化代码时删了 `restored_chunks = 0` 但 return 语句还在引用。通过 uvicorn 输出日志找到 traceback。

**教训**：改代码要全文搜索变量引用，确保定义-使用链路完整。

## 问题 14：快照回滚后文件未恢复——`delete_file` 默认值

**日期**：2026-07-20

**现象**：创建快照 → 删文件 → 回滚成功 → 文件数没变。

**排查**：`file` 表中文件记录已被删除。追踪到 `remove_file_from_knowledge` 端点的 `delete_file: bool = Query(True)`——默认连源文件一起删。快照只存 file_id 引用。

**解决**：新增 **Unlink Only** 按钮（`?delete_file=false`），橙色标识，只取消关联保留文件。涉及 3 个文件修改：API 客户端、Files 组件、KnowledgeBase 页面。

**教训**：下游 API 默认参数是隐藏的坑。增量功能的边界条件要写清文档。

---

## 问题 15：独立脚本能跑，API 500 无 JSON——如何定位进程级崩溃

**日期**：2026-07-20

**现象**：`python -c` 独立脚本测试通过，FastAPI 端点返回空响应+500。

**关键方法**：FastAPI 的 Python 进程级崩溃（NameError 等）不会返回 JSON。排查方法是读取 uvicorn 后台任务输出文件中的 stderr traceback。

**教训**：API 500 != HTTPException。进程崩溃时看服务端日志，不是看 HTTP response。
