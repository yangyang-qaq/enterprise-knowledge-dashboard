# Python Web 后端开发最佳实践

## 一、项目结构

一个清晰的项目结构是工程化的基础。推荐使用以下分层架构：

```
project/
  src/
    api/          # 路由层 - 处理 HTTP 请求和响应
    services/     # 业务逻辑层
    models/       # 数据模型层
    repositories/ # 数据访问层
    utils/        # 工具函数
    config/       # 配置管理
  tests/          # 测试代码
  migrations/     # 数据库迁移
  scripts/        # 运维脚本
```

分层的好处是每层职责清晰，便于测试和维护。路由层只负责参数验证和响应格式化，业务逻辑放在 services 中，数据操作通过 repositories 抽象，这样可以在不修改业务逻辑的情况下更换数据库。

## 二、异步编程

Python 的 asyncio 生态日益成熟。FastAPI 是目前最流行的异步 Web 框架，其性能接近 Node.js，显著优于传统的 Flask 和 Django。

使用 async/await 时需要注意：不要在异步函数中调用同步阻塞操作（如 requests 库），否则会阻塞事件循环。应该使用 httpx 等异步库替代，或者用 asyncio.to_thread 将同步操作放到线程池中执行。

数据库方面，SQLAlchemy 2.0 提供了完善的异步支持。通过 AsyncSession 和 async 查询方法，可以在不修改业务逻辑的情况下获得异步性能。但需要注意惰性加载在异步上下文中会失败，应该使用 selectinload 或 joinedload 预加载关联数据。

## 三、依赖注入

依赖注入是构建可测试、可维护后端服务的核心模式。FastAPI 内置了强大的依赖注入系统——Depends，可以将认证、数据库会话、配置等作为依赖项注入到路由函数中。

实践中应该避免创建全局的单例对象。将外部依赖（数据库连接、Redis 客户端、外部 API 客户端）通过依赖注入的方式传递给业务代码，使得单元测试时可以轻松替换为 mock 对象。

## 四、类型注解与数据验证

Python 3.12+ 的类型注解系统已经非常成熟。结合 Pydantic 和 mypy，可以在开发阶段捕获大量类型错误。Pydantic v2 采用 Rust 内核重写，性能提升了 5-50 倍。

数据验证应该发生在系统边界——API 层的输入验证和数据库层的输出验证。Pydantic 的 Field 验证器可以定义自定义验证逻辑，model_validator 可以实现跨字段验证。

建议为所有 API 的输入输出定义明确的 Pydantic 模型，不要使用 dict 或 Any 类型。这不仅提高了代码安全性，也为 API 文档（OpenAPI/Swagger）提供了准确的 Schema。

## 五、错误处理

统一的错误处理策略对 API 的可用性至关重要。应该定义全局异常处理器，捕获未经处理的异常并转换为标准化的错误响应格式。

业务异常应该继承自自定义的基类（如 AppException），包含错误码、错误消息和 HTTP 状态码。在异常处理器中根据异常类型映射到对应的 HTTP 响应。

FastAPI 的 HTTPException 适用于简单的场景，但对于复杂的业务错误，建议使用自定义异常类配合全局异常过滤器。

## 六、测试策略

测试金字塔仍然是有效的指导原则：大量单元测试 + 适量集成测试 + 少量端到端测试。

单元测试应该覆盖核心业务逻辑。使用 unittest.mock 或 pytest-mock 来模拟外部依赖。使用 pytest-asyncio 可以方便地测试异步代码。

集成测试需要验证各层的协作是否正确。使用测试数据库（SQLite :memory: 或 Docker 中的 PostgreSQL）来隔离测试环境。pytest 的 fixture 系统非常适合管理测试依赖。

端到端测试验证完整的业务流程。对于 API 服务，可以使用 httpx 或 requests 对运行中的服务发送真实请求。关键路径（注册、登录、核心业务流程）必须被端到端测试覆盖。

## 七、性能优化

数据库查询是最常见的性能瓶颈。N+1 查询问题可以通过预加载关联数据解决。对于复杂查询，可以使用 EXPLAIN ANALYZE 分析执行计划，添加合适的索引。

缓存策略应该根据数据特性选择。频繁读取但很少变更的数据适合用 Redis 缓存；需要实时计算的数据可以考虑用异步任务预计算。

API 响应压缩（Brotli/Gzip）可以显著减少网络传输量。对于大型列表接口，分页是强制性的。对于实时性要求不高的查询，可以考虑使用 ETag 或 Last-Modified 实现条件请求，减少不必要的数据传输。
