# 微服务架构设计指南

## 第一章：微服务概述

微服务架构是一种将单一应用程序划分为一组小型、独立服务的设计风格。每个服务运行在自己的进程中，通过轻量级通信机制（通常是 HTTP/ RESTful API 或消息队列）互相协作。这些服务围绕业务能力构建，可以由不同的团队使用不同的编程语言、框架和数据存储技术独立开发和部署。

微服务架构的核心理念来源于 Unix 哲学——"做好一件事"。每个微服务应该只负责一个特定的业务领域，拥有明确的边界，并且可以独立演进和扩展。

## 第二章：服务拆分策略

服务拆分是微服务架构设计的最大挑战。拆分得太粗，就失去了微服务的优势；拆分得太细，则会导致服务间通信开销过大、运维复杂度剧增。

领域驱动设计（DDD）是指导微服务拆分的核心方法论。通过识别业务领域中的限界上下文（Bounded Context），可以自然地定义服务边界。每个限界上下文对应一个微服务，拥有自己独立的数据模型和业务逻辑。

常见的拆分模式包括：按业务能力拆分、按子域拆分（核心域、支撑域、通用域）、按团队组织拆分（康威定律）。在实践中，往往需要综合运用多种模式。

## 第三章：服务间通信

微服务之间的通信方式主要分为两种：同步通信和异步通信。

同步通信通常使用 HTTP/REST 或 gRPC。REST 简单直观，适合对外 API；gRPC 基于 HTTP/2 和 Protocol Buffers，性能更好，适合内部服务间的高频调用。

异步通信通过消息队列或事件流实现，如 RabbitMQ、Kafka、AWS SQS 等。事件驱动架构是推荐的异步通信模式——服务发布领域事件，其他服务订阅并响应，实现松耦合。

选择通信方式的决策矩阵：请求-响应模式用同步，需要解耦或削峰填谷用异步，跨服务事务用 Saga 模式配合异步消息。

## 第四章：数据管理

微服务架构中最重要的原则之一是每个服务拥有自己的数据库。这意味着服务不能直接访问其他服务的数据库，必须通过 API 进行数据交换。

数据一致性是分布式系统中的经典难题。ACID 事务在单体数据库中容易实现，但在分布式环境下代价极高。最终一致性成为更实际的目标。

Saga 模式是实现分布式事务的主流方案。每个服务执行本地事务并发布事件，下一个服务监听事件并执行自己的本地事务。如果某一步失败，前面的步骤通过补偿事务回滚。

CQRS（命令查询职责分离）和事件溯源（Event Sourcing）是处理复杂查询场景的有效模式。通过将读写模型分离，可以分别优化命令端和查询端的性能。

## 第五章：服务治理

服务发现和负载均衡是微服务架构的基础设施。客户端发现模式（如 Netflix Eureka）和服务端发现模式（如 Kubernetes Service）各有优劣。

API 网关是微服务架构的入口，负责请求路由、认证鉴权、限流熔断、日志监控等横切关注点。Kong、APISIX、Spring Cloud Gateway 是常用的开源方案。

熔断器模式（Circuit Breaker）是防止级联故障的关键。当某个服务出现故障时，熔断器快速失败，避免请求堆积和资源耗尽。Resilience4j 和 Sentinel 是 Java 生态的主流选择。

## 第六章：可观测性

在微服务架构中，一个用户请求可能跨越多个服务，传统的日志和监控方式已经不够用了。可观测性的三大支柱是：日志（Logging）、指标（Metrics）和链路追踪（Tracing）。

ELK Stack 和 Grafana Loki 是集中式日志管理的流行方案。Prometheus + Grafana 是云原生环境中指标监控的事实标准。Jaeger 和 Zipkin 是分布式链路追踪的常用工具。

OpenTelemetry 正在成为可观测性数据采集的统一标准，它提供了一套与厂商无关的 API、SDK 和工具，支持多种语言，可以同时生成 Trace、Metric 和 Log 数据。
