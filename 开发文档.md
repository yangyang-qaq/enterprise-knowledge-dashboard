## Phase 1：分块预览与手动调整

**完成日期**：2026-07-20

**解决的问题**：用户上传文档后无法看到文档如何被分块，分块质量无法验证，切坏了也不知道。

**实现的功能**：
- 文档上传后自动展示所有分块（编号、内容预览、token 数、元数据）
- 支持合并相邻分块
- 支持在指定位置拆分量块
- 调整完成后重建向量索引

---

### 1.1 数据库设计

**新建表**：`knowledge_chunk`

**Alembic 迁移文件**：[a7b8c9d0e1f2_add_knowledge_management_tables.py](backend/open_webui/migrations/versions/a7b8c9d0e1f2_add_knowledge_management_tables.py)

```sql
CREATE TABLE knowledge_chunk (
    id            TEXT PRIMARY KEY,           -- UUID
    knowledge_id  TEXT NOT NULL,              -- 所属知识库 FK
    file_id       TEXT NOT NULL,              -- 所属文件 FK
    chunk_index   BIGINT NOT NULL,            -- 分块序号（0-based）
    content       TEXT NOT NULL,              -- 分块完整文本
    token_count   BIGINT,                     -- 估算 token 数
    meta          JSON,                       -- 元数据 {page, section_header, ...}
    content_hash  TEXT,                       -- SHA-256 去重/对比用
    created_at    BIGINT NOT NULL,
    updated_at    BIGINT NOT NULL,

    UNIQUE(file_id, chunk_index),
    FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE,
    FOREIGN KEY (file_id) REFERENCES file(id) ON DELETE CASCADE
);

CREATE INDEX ix_knowledge_chunk_knowledge_id ON knowledge_chunk(knowledge_id);
CREATE INDEX ix_knowledge_chunk_file_id ON knowledge_chunk(file_id);
```

> 同时创建了 Phase 2-4 的 4 张表（knowledge_processing_task, knowledge_batch_task, knowledge_relevance_judgment, knowledge_snapshot），详见迁移文件。

### 1.2 后端 API

**新增端点**（全部挂载在 `/api/v1/knowledge/{id}/` 下）：

#### POST `/{id}/chunks/preview` — 分块预览

不写入向量库，仅运行 loader + splitter 展示分块结果。

**请求体**：
```json
{
  "file_id": "xxx-xxx-xxx"
}
```

**返回**：
```json
[
  {
    "id": "uuid",
    "knowledge_id": "xxx",
    "file_id": "xxx",
    "chunk_index": 0,
    "content": "第一章内容...",
    "token_count": 245,
    "meta": {"page": 1, "section_header": "## 引言"},
    "content_hash": "sha256...",
    "created_at": 1234567890,
    "updated_at": 1234567890
  }
]
```

**处理流程**：
```
1. 查找 file_id → 获取文件路径
2. Storage.get_file() → 本地文件
3. build_loader_from_config() → Loader 加载文档（PDF/DOCX/...）
4. loader.aload() → 提取文本
5. MarkdownHeaderTextSplitter (可选) → 按标题分块
6. RecursiveCharacterTextSplitter / TokenTextSplitter → 按 chunk_size 分块
7. 写入 knowledge_chunk 表（先删旧记录，再插入新记录）
8. 返回分块列表给前端
```

**代码位置**：[backend/open_webui/routers/knowledge.py](backend/open_webui/routers/knowledge.py) — `preview_knowledge_chunks()` 函数

#### POST `/{id}/chunks/merge` — 合并分块

**请求体**：
```json
{
  "knowledge_id": "xxx",
  "file_id": "xxx",
  "start_index": 3,
  "end_index": 5
}
```

**处理流程**：
```
1. 读取 chunk_index 在 [start, end] 范围内的分块
2. 内容拼接（用 \n\n 分隔）
3. 创建新分块插入（chunk_index = start_index）
4. 删除原有分块
5. 后面的分块重新编号（chunk_index -= (end - start)）
```

> **约束**：只能合并相邻分块（chunk_index 连续），前端会校验。

---

#### POST `/{id}/chunks/split` — 拆分分块

**请求体**：
```json
{
  "chunk_id": "xxx",
  "split_at": 200     // 在第 200 个字符处拆分
}
```

**处理流程**：
```
1. 读取 chunk_id
2. 在 split_at 位置切分为 part_a 和 part_b
3. 原分块更新为 part_a
4. 插入新分块 part_b（chunk_index = 原 index + 1）
5. 后续分块下标 +1
```

> 约束：split_at 必须在 1 到 content.length-1 之间，且两部分都非空。

---

#### POST `/{id}/chunks/reindex` — 重建向量

**处理流程**：
```
1. 读取 knowledge_chunk 表中该 KB 的所有分块
2. 删除该 KB 在 ChromaDB 中的旧集合
3. 将分块转为 LangChain Document 列表
4. 调用 save_docs_to_vector_db() 重新生成向量 + 入库
```

## Phase 2：向量化进度可视化

**完成日期**：2026-07-20

**解决的问题**：用户批量上传文档后无法知道处理进度，文件是卡住了还是正在处理，完全不可见。

**实现的功能**：
- 每个文件独立展示处理状态（pending → chunking → embedding → completed / failed）
- 实时进度条（百分比 + chunks 进度）
- SSE (Server-Sent Events) 实时推送 + 3 秒轮询兜底
- 整体进度汇总（已完成/总数/失败数）
- 处理失败时展示错误信息

---

### 2.1 数据库设计

CREATE TABLE knowledge_processing_task (

## Phase 3：检索质量评估面板

**完成日期**：2026-07-20

**解决的问题**：无法评估知识库的检索质量——检索结果相关吗？召回率多少？没有人知道。

**实现的功能**：
- 输入测试查询 → 返回 Top-K 检索结果 + 相似度分数
- 标注每个结果「相关 ✓」或「不相关 ✗」
- 基于标注自动计算 recall@K、precision@K、MRR 指标
- 查询历史记录 + 标注持久化

---

### 3.1 后端 API

#### POST `/{id}/evaluate/query` — 执行评估查询

调 `query_collection()` 检索 → 返回格式化结果 + 指标

#### POST `/{id}/evaluate/annotate` — 标注检索结果

Upsert 标注到 `knowledge_relevance_judgment` 表

#### GET `/{id}/evaluate/judgments` — 查看历史标注

按 query_text 分组返回

#### DELETE `/{id}/evaluate/judgments/{query_text}` — 删除标注

---

### 3.2 关键技术细节

**调用 `query_collection` 的正确方式**（问题 #11）：
```python
result = await query_collection(
    request=request,
    collection_names=[id],           # list, not str
    queries=[form_data.query],        # list, not str
    embedding_function=request.app.state.EMBEDDING_FUNCTION,  # required
    k=k,
)
# 返回格式: {'documents': [[]], 'distances': [[]], 'metadatas': [[]]}
```

**指标计算**（服务端 Python）：
- `recall@K` = 检索到的相关文档数 / 总相关文档数
- `precision@K` = 检索到的相关文档数 / K
- `MRR` = max(1 / rank_of_relevant) 跨所有标注

## Phase 4：知识库版本管理

**完成日期**：2026-07-20

**解决的问题**：增删文件后无法回退到之前的知识库状态，配置错误无法撤销。

**实现的功能**：
- 创建快照（保存当前所有文件关联 + 分块记录的元数据）
- 快照卡片列表（标签、文件数、分块数、时间）
- 一键回滚（清除当前状态 → 恢复快照中的文件关联 → 重新向量化）
- 两个快照差异对比（新增/删除/修改的文件）
- 删除快照

---

### 4.1 后端 API

#### POST `/{id}/snapshots` — 创建快照

收集 `knowledge_chunk` 表中所有分块 + `knowledge_file` 关联 → JSON 序列化存入 `snapshot_data`。

#### GET `/{id}/snapshots` — 快照列表

按时间倒序返回。

#### POST `/{id}/snapshots/{sid}/rollback` — 回滚

1. 清除当前 chunk 记录和 file 关联
2. 从 snapshot_data 恢复分块元数据
3. 恢复文件关联（仅恢复元数据，向量需通过 Chunks Tab 手动 Reindex）
4. 返回恢复统计 + 提示信息

**设计权衡**（问题 #14）：快照只保存 file_id 引用而非文件内容。如果文件被物理删除（默认 `delete_file=True`），回滚无法恢复。为此新增 **Unlink Only** 按钮（`?delete_file=false`），只取消关联保留文件，使回滚可正常工作。

#### POST `/{id}/snapshots/compare` — 对比

对比两个快照的 file_id 集合差异，检查同一 file_id 的 content_hash 是否变化。

#### DELETE `/{id}/snapshots/{sid}` — 删除

RAG 注入流程：
